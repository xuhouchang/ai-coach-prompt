# AI Coach Prompt：面向小白的交互式 Prompt 学习 Skill

> 不只是“看懂 Prompt”，而是在一次次互动中真正学会把模糊需求写成可执行的 AI 指令。

AI Coach Prompt 是一个面向初学者的教学型 Skill。它不替用户完成业务任务，也不替用户直接执行 Prompt；它的职责是陪用户把一个模糊、口语化的需求，逐步拆解、诊断并改写成更清晰、更可复用的 AI Prompt。

它适合刚开始使用 AI、知道“要写 Prompt”但不知道从哪里下手的人。用户不需要先看完长视频或读完一套教材，而是可以直接在交互中练习：回答问题、观察示例、诊断缺口、完成改写、获得反馈，再把能力迁移到新的场景里。

## 我们的初衷：Learn in Practice

我们相信，单纯看视频、读文档，往往不足以真正形成能力。

尤其在 AI 时代，很多能力并不是“记住一套知识”就够了，而是要在具体问题里反复判断：

- 这个需求到底要 AI 做什么？
- 缺的是输入、背景、输出，还是验收标准？
- 一句话需求为什么会让 AI 不断猜测？
- 怎样把一次性的表达，变成下次也能复用的 Prompt？

所以我们尝试把学习设计成一种交互式练习：

```text
看见问题 → 自己判断 → 动手改写 → 得到针对性反馈 → 再练一次 → 迁移到新任务
```

这就是 **Learn in Practice**：学习不是被动接收，而是在真实的思考和练习中形成可迁移的能力。

## 这个 Skill 能做什么

AI Coach Prompt 会带用户完成一条循序渐进的 Prompt 学习路径：

1. **快速诊断**  
   通过简短问题了解用户的经验、行业和学习目标，匹配合适的练习场景。

2. **建立核心认知**  
   用简单的方式理解：Prompt 不是“随便说一句话”，而是一份让 AI 少猜测的任务说明。

3. **识别任务和信息缺口**  
   引导用户分辨任务类型，并检查一个需求是否说明了：
   - 要做什么
   - 基于什么材料
   - 给谁使用
   - 交付什么结果
   - 怎样才算做好

4. **亲手改写 Prompt**  
   用户自己完成第一版改写，而不是直接拿到答案。

5. **获得结构化反馈并再练一轮**  
   Skill 会指出最关键的 1–2 个缺口，帮助用户把 Prompt 改得更稳定、更可执行、更方便复用。

6. **迁移到新场景**  
   在不同类型的需求上再做一次练习，避免只会套一个模板。

## 它和普通 Prompt 模板有什么不同

普通模板给用户一个“可以复制的答案”；AI Coach Prompt 更关注用户为什么会写、下次能不能自己写。

它的特点包括：

- **对小白友好**：从短问题和实际例子开始，不要求用户一上来就写长 Prompt。
- **一轮只做一件事**：减少信息负担，让用户始终知道下一步该做什么。
- **强调练习与反馈**：不是只展示好 Prompt，而是让用户亲自诊断和改写。
- **根据场景调整**：可以结合用户的行业和任务类型选择练习案例。
- **兼顾可执行性与复用性**：不仅让 Prompt 更清楚，也让它更适合下次继续使用。
- **安全边界明确**：这是教学 Skill，不替用户执行任务；对无关、危险或不合规请求会拒绝并保持在学习范围内。
- **跨平台设计**：核心内容不依赖特定宿主能力，支持有卡片/组件时的增强展示，也提供纯文本回退。

## 支持的平台

当前 Skill 特别适配以下平台：

- **腾讯 WorkBuddy**：支持交互式学习流和可视化增强展示。
- **Trae**：支持 Markdown 形式的学习与练习。
- **Coze（扣子）**：可映射为卡片和工作流节点。
- **Codex 及其他支持 SKILL.md 的宿主**：可通过纯文本方式完成完整学习流程。

不同平台的呈现方式可能不同，但核心学习流程保持一致：**诊断、练习、反馈、迁移**。

## 适合谁

- 刚开始使用 ChatGPT、WorkBuddy、Trae 或 Coze 的用户
- 经常觉得“AI 怎么总是理解错我”的用户
- 会写几句 Prompt，但输出总是不稳定的用户
- 希望把真实业务需求说清楚、写清楚的产品、运营、市场、销售、研究和职能团队成员
- 希望把 AI 使用从“偶尔试试”变成“可重复工作方法”的团队

## 使用方式

将完整 Skill 文件夹导入支持的宿主平台，并以 `SKILL.md` 作为入口。

请保留目录中的配置、课程内容、模块、适配器和资源文件；不要只单独复制 `SKILL.md`，否则互动流程和平台适配可能不完整。

## 如何安装与使用

这个 GitHub 链接可以作为 Skill 的**分发入口**：把仓库链接交给一个具备 GitHub 访问权限和本地文件操作权限的 Agent，它可以下载完整目录，再按宿主平台完成安装。

但请注意：**链接本身不等于已经安装。** 无论在哪个平台，都必须保留完整 Skill 文件夹，而不是只复制 `SKILL.md` 或 README。

### 交给 Agent 的通用说法

可以把下面这段话连同本仓库链接一起发给 Agent：

> 请从这个 GitHub 仓库获取完整的 AI Coach Prompt Skill。不要只读取 README 或 SKILL.md；请保留仓库根目录下的全部 Skill 文件，并按你当前宿主平台的 Skill 安装方式导入。安装完成后，以 SKILL.md 作为入口启动一次教学流程。

### 腾讯 WorkBuddy

对于腾讯 WorkBuddy，Agent 应下载或克隆完整 Skill 文件夹，并通过包内的 `workbuddy_install_manifest.yaml` 作为安装契约导入。

推荐做法是把**完整文件夹**提供给 WorkBuddy，而不是逐个上传 YAML 或 Markdown 文件。支持 GitHub 读取的 Agent 可以先拉取仓库，再执行 WorkBuddy 的导入动作。

### Trae

Trae 中可将完整目录复制到项目的：

```text
.trae/skills/ai-coach-prompt/
```

也可以放入 Trae 配置的全局 Skills 目录。关键是 `SKILL.md` 必须位于该 Skill 的根目录。

### Coze（扣子）

Coze 需要上传或打包完整 Skill 目录，并保持 `SKILL.md` 位于根目录。GitHub 链接可以让 Agent 获取源文件，但 Agent 仍需将完整目录导入 Coze；不能只把链接当作 Coze Skill 的运行入口。

### 其他支持 SKILL.md 的 Agent 或宿主

只要 Agent 能访问 GitHub、下载目录并安装本地 Skill，就可以把本仓库链接作为安装来源。没有可视化组件时，Skill 会以 Markdown/纯文本方式继续教学流程。

## 未来方向

AI Coach Prompt 是我们交互式学习课程的起点。

后续我们计划继续探索更多通过 Skill 完成的课程：不把学习局限于视频和文档，而是让用户在真实任务中，通过交互、练习、反馈和迭代来获得能力。未来的主题可能会延伸到：

- 更复杂的 Prompt 设计与任务拆解
- Workflow 与多步骤任务设计
- Skill 的设计与封装
- MCP 与工具调用的基础认知
- Agent 工作方式与协作设计

我们希望每一门课都遵循同一个原则：

> **让学习发生在实践里，而不是只停留在观看和阅读里。**

## 说明

本项目是教学型 Skill。它帮助用户学习如何写出更清晰、更安全、更可执行的 Prompt；它不会代替用户执行业务任务，也不会提供危险、违法或不合规内容的帮助。
